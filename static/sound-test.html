forya iOS sound-carry test harness

Run on the operator's iPhone, Safari (the device that reproduces the bug). Each section is independent. Tap BLESS first where noted — that one tap is the only legal gesture. Then run the no-gesture steps and watch the readout: a green AUDIBLE = stayed playing & unmuted; a red PAUSED/MUTED = iOS re-gated (our bug). The objective signal auto-detects the pause failure; for the subtler "plays-but-silent" case use the 🔊 heard/🔇 silent ear-confirm buttons. Copy the LOG (bottom) or the export box at the end and paste it back to the swarm.

0 · Config — set 3 real clips WITH AUDIO from best

Default points at relative /api/media/<name> (works if this file is served from the same host as a forya instance). Otherwise edit to absolute https://<best-host>/api/media/<name>. Clips MUST have an audio track or the audio-gate tests are meaningless. Cross-origin is fine for every test EXCEPT the optional Web-Audio RMS meter (needs same-origin/CORS).

A · POOL src-swap carry — THE linchpin (audio gate, Option 1)

One persistent <video>. BLESS once (gesture) → then swap src + play() with NO further gesture, repeatedly (this is also the auto-advance case, matrix ii). Question: does the blessing survive the src-swap for audible output? If yes → pooling is the real fix and auto-advance-with-sound works.

The ⏱️ delayed case is the decisive one: it separates a durable per-element blessing (pool carries a minutes-long session) from a ~5s sticky-activation window (pool useless). After it blesses, do NOT touch the screen until the countdown ends — any tap re-arms activation and voids the test.

B · scrollend unmute on FRESH-per-card elements (Option 4 probe)

Mirrors forya's CURRENT architecture: a fresh <video> per card in a real scroll-snap container. BLESS once, then scroll — on each scrollend (post-snap, inside sticky activation) we set muted=false on the settled card's element with no accompanying play(). If the settled card stays audible → dev's "synchronous-in-gesture-call-stack" was too strong (activation-scoped wins, cheap fix). If it pauses → strict-synchronous holds → need the pool.

C · Multi-bless in one gesture (Option 2 bridge)

Documented iOS trick: inside ONE click handler, unmute several elements — including ones that haven't played yet. Then play a not-yet-played one LATER with no gesture. Audible? → the windowed multi-bless bridge is viable (carries on scroll, but not auto-advance).

D · Off-gesture muted→unmuted toggle (matrix iii)

On the blessed pool element (section A), flip muted false→true→false with no gesture. Does the false-transition re-pause? (pool keeps neighbors muted, unmutes on activation.)

E · Seek / Range during audible playback (matrix iv)

On the blessed pool element, set currentTime to mid-file (fires an HTTP Range request — forya's serving-four contract). Does audio continue uninterrupted?

F · DOM re-parent of a playing blessed element (matrix v)

The pool repositions elements across cards. Move the blessed pool <video> to a different DOM parent and back — does it keep playing/audible/blessed (no reload, no pause)?
